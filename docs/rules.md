# Documentation Rules

These rules should be followed while writing the PayGuard dissertation documentation.

## Figures and Screenshots

- Do not insert actual figures, diagrams, or screenshots yet.
- When a figure is required in the main report, use only a short placeholder such as `Figure 3.1: Use case diagram`.
- Store full diagram and screenshot descriptions in `docs/figure-descriptions.md`.
- The actual figures, diagrams, and screenshots will be added later after the written structure is approved.

## Writing Style

- Keep the language simple, academic, and easy to defend during presentation.
- Avoid overly technical explanations where a simple explanation is enough.
- Explain technical terms before using them heavily.
- Write as a final year project report, not as developer documentation.

## Project Scope

- Treat PayGuard as a prototype system.
- Mention that synthetic data is used for privacy and academic development.
- Do not claim that the system is connected to a real bank, university payment system, or live mobile money provider.
- Describe the system as decision support, not as a tool that automatically proves fraud.

## Consistency

- Use the project name `PayGuard` consistently.
- Use `mobile money fraud detection system` as the main project description.
- Keep chapter numbering consistent.
- Keep tables in Markdown for now so they can be converted to Word tables later.
